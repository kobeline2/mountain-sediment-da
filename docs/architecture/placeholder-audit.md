# プロジェクト理解メモ

## 1. 研究の中心テーマ

この修論の中心は、次の 2 本立てです。

1. ハイドロフォンの複数倍率パルス数から、掃流砂量とその不確実性を推定する
   ガウス過程回帰モデルを作る。
2. その推定結果を、山地流域の土砂動態シミュレーションの境界条件評価
   あるいはデータ同化に活用する。

修論要旨と本文の構成から見ると、論文としての主張は以下の流れです。

- 第 3 章: 流砂量推定モデルの構築
- 第 4 章: 構築したモデルの適用と検証
- 第 5 章: シミュレーション結果と照合してデータ同化への活用を考察

### ガウス過程回帰による流砂量推定

- `修論データ/ガウス過程計算/2023_mt_ks`

このディレクトリが、現時点で最も「中核コード」に近いです。
構造は比較的きれいで、役割も明確です。

- `main.m`
  - 学習モードと予測モードの切り替え
- `parameters.m`
  - 使うデータ、目的変数、入力列、学習済みパラメータ、出力先の定義
- `init.m`
  - パス設定と GPML 初期化
- `src/preprocessing`
  - 元データから学習入力を作る
- `src/utils`
  - train/test 分割、正規化、Excel 出力など
- `src/models`
  - `gpval.m`: ハイパーパラメータ学習
  - `gpfit.m`: 推論
- `src/postprocessing`
  - 平均、標準偏差、逆正規化、図化用整形
- `data/raw`, `data/processed`
  - 元 Excel と前処理済み MAT
- `res/prediction`
  - 予測 Excel

実装上の流れは、ほぼ次の一本線です。

`data -> preprocess -> split -> normalize -> GP学習/推論 -> postprocess -> Excel出力`

### シミュレーション・データ同化側

- `修論データ/データ同化simhis`

このディレクトリは 10GB 以上あり、現在の混乱の主因です。
中には `c0`, `ck1`-`ck6`, `cs1`-`cs6` などのケースディレクトリがあります。

調べた範囲では、各ケースに含まれる以下の主要ソースは全ケースで同一でした。

- `simhis.f90`
- `para.f90`
- `sedsupply_data.f90`
- `makefile`

つまり、ここは「ケースごとに計算エンジンを丸ごと複製している」構成です。
差分は主にケースごとの入力条件で、特に `conditions/sedpro_depth.dat` が
ケース差分の中心になっています。

確認できた範囲では、ケース差分はかなり限定的です。

- `c0`
  - 土砂供給なし
- `ck1`-`ck6`
  - 同じ 2 セルに対して段階的に供給量を増やす系列
- `cs1`-`cs6`
  - `ck*` とは別の 2 セルに対して段階的に供給量を増やす系列

少なくとも今回見た `sedpro_depth.dat` では、
各ケースとも非ゼロ入力は 1 時刻だけで、しかも 2 セルだけでした。
つまり、論文第 5 章の比較は
"共通シミュレーションエンジンに対し、供給位置と供給量だけを変えたケース比較"
と読めます。

Fortran 側の概念モデルは以下です。

- `conditions/*.dat`
  - 降雨、河道、粒径、土砂供給条件などの入力
- `watershed_data/*.dat`
  - 流域・河道形状などの静的データ
- `simhis.f90`
  - 計算本体
- `results/*.dat`
  - 流量、流砂量、河床高、累積供給量などの出力

`open_files.f90` を見ると、入力と出力の境界は比較的明確です。
なので、アーキテクチャ上はかなり整理しやすいです。

## 3. 現状のアーキテクチャ上の問題点

### 1. コードとデータと成果物が混在している

- 学習コード
- 生データ
- 前処理済みデータ
- 推定結果
- シミュレーション結果
- 図表作成途中の Excel
- 論文原稿

これらが同じ階層に近い場所で共存しています。

### 2. シミュレーションケースごとにコードが重複している

`データ同化simhis` は、本来は

- 共通エンジン
- ケースごとの入力
- ケースごとの結果

に分けられるはずですが、今はケースごとにエンジンごと複製されています。

### 3. 正式版と旧版の境界が曖昧

`ガウス過程計算` の中には以下が混在しています。

- `2023_mt_ks`
- `旧型ガウス計算結果/*`
- GPML の複数コピー

このため、
"今後触るべき本体はどれか"
が初見で判断しづらいです。

### 4. 実行再現に必要な設定がコードの外に散っている

MATLAB 側では以下のような環境依存の前提があります。

- `config.m`
- `parameters.m`
- GPML のローカルパス
- `res/param/240123sato/*`

研究室内では動いても、引き継ぎや論文化の追試には不利です。

### 5. 論文の主張とファイルの対応関係が見えにくい

修論本文は比較的きれいに整理されていますが、
どの章のどの図がどのコード・どの出力に対応するかは
フォルダだけ見てもすぐにはわかりません。

## 4. 現時点での「正」として扱えそうなもの

完全確定ではありませんが、今後の整理の出発点としては以下が有力です。

### 流砂量推定モデル本体

- `修論データ/ガウス過程計算/2023_mt_ks`

理由:

- 構造が最も整理されている
- README がある
- `src` 配下の責務分割が明確
- `parameters.m` に実際の使用痕跡がある
- `.git` があり、独立プロジェクトとして扱われていた形跡がある

補足:

- 現在の `parameters.m` は `res/param/240123sato/*` を参照しており、
  学習済みモデルを使って `data/processed/20180625.mat` を予測する設定になっている
- つまり、学習済みパラメータを固定して個別期間を再評価する運用が
  すでに行われている

### シミュレーション本体

- `修論データ/データ同化simhis` のうち、任意の 1 ケースのソース一式

理由:

- 主要 Fortran ソースはケース間で同一
- ケース差分は主に入力ファイル

つまり、今後は
"代表ケースから engine を切り出し、各ケースは scenario として持つ"
のが自然です。

## 6. この構成にすると何が良くなるか

### 論文を書くとき

- どの図表がどの出力から来たか追いやすい
- 修論から論文への再構成がしやすい
- 研究の主張と実装の対応が明瞭になる

### コードを再実行するとき

- MATLAB 側と Fortran 側の責務が分かれる
- ケース追加が「ディレクトリ複製」ではなく「scenario 追加」になる
- 結果だけ消して再計算する、といった操作が安全になる

### 引き継ぎのとき

- "どれが最新か" を説明しやすい
- 参照専用の旧版を archive に追い出せる
- 実験データと生成物の境界が明確になる

## 7. 次にやるとよいこと

優先度順だと、次の順番が良いです。

1. `2023_mt_ks` を流砂量推定モデルの正本とみなすか確認する
2. `データ同化simhis` から共通 engine を 1 つ切り出す
3. ケース差分を `sedpro_depth.dat` などの scenario 入力に集約する
4. `3.3`, `4.1`, `4.2`, `5.1`, `5.2` を「論文図表用データ」として位置づける
5. 論文本文の各図表と、生成元スクリプト・元データの対応表を作る

## 8. いまの理解での暫定結論

このプロジェクトは「壊れている」というより、
"研究の過程で自然に増えたものが、まだ整理されていない" 状態です。

なので、最初にやるべきことは全面的な書き換えではなく、

- 正本の決定
- engine と scenario の分離
- code / data / outputs / paper の分離

です。

この 3 点ができると、修論から論文へのブラッシュアップ作業はかなり進めやすくなります。
